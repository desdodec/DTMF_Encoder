Developer Specification: Web-Based Registration, Local Storage DB, and Automated DTMF Pipeline1. Project OverviewThe goal is to implement a responsive, lightweight single-page web utility into the https://github.com/desdodec/DTMF_Encoder.git repository. This tool will streamline field data collection for a generative art project.The interface must allow an administrator to input a subject's first name, automatically log the session metadata (ID, Name, Timestamp) to a secure local database, and fire a precisely timed DTMF audio sequence matching the subject's unique ID directly into an audio recording setup.2. Technical Requirements & Specifications3.1 Front-End / UI LayerLanguage: Semantic HTML5, CSS3 (preferably responsive utility classes like Tailwind, or clean embedded Flexbox/Grid for cross-device mobile support).Predictive Input: Implement a <input type="text"> binding linked to an easily expandable <datalist> containing common regional first names to accelerate field data entry.State Management: The UI must display an absolute index of the next registration profile (e.g., "Ready for Subject 008") to give the field coordinator immediate contextual status.Console / Output Log: A scrollable, reverse-chronological status feed embedded on-screen to confirm successful DB commits and show the exact DTMF payload sequence being generated.3.2 Database Layer (Local Prototyping)Storage Engine: Standard localStorage API (serialized via JSON) or IndexedDB for persistence across accidental browser refreshes.Schema Definition:TypeScriptinterface SubjectRecord {

}
Data Portability: * Provide a dedicated administrative utility button to parse the local data array into a standard Comma-Separated Values (.csv) string.Trigger an automated browser download anchor containing the full table payload, formatted as: gait_session_export_[YYYY-MM-DD].csv.Provide a destructive "Reset Session Data" hook with an explicit browser confirmation dialog (confirm()) to prevent accidental data erasure.3.3 Audio Synthesis Layer (The DTMF Engine)API: Native Web Audio API (AudioContext). No external heavy audio frameworks allowed.Bypassing Mobile Sandboxing: Ensure AudioContext is instantiated or resumed only inside a synchronous user gesture callback (i.e., the click handler for the registration submit button) to prevent mobile browser security policy blocks.DTMF Matrix Specification:Standard 4x4 matrix mapping must be defined using pairs of clean, mathematical sine wave oscillators:Frequency1209 Hz1336 Hz1477 Hz1633 Hz697 Hz123A770 Hz456B852 Hz789C941 Hz*0#DTransmission Framing Pipeline:Start Delimiter: *Payload: Auto-incremented, zero-padded 3-digit subject ID string (e.g., 004)End Delimiter: #Example structural payload for Subject 4: *004#Timing Constraints:To guarantee automated parsing software (like Python's scipy or dtmf packages) can cleanly isolate the transient frequency blocks from background pavement or wind noise, use the following strict timing window:Tone Duration: Exactly 80ms (0.08 seconds)Inter-Tone Gap / Pause: Exactly 30ms (0.03 seconds)Gain Volume Control: Standardize gain profile output to 0.25 maximum to eliminate harmonic distortion on small mobile speakers.4. Acceptance Criteria & Edge CasesiOS/Safari Compliance: The interface must produce loud, audible, dual-sine tones on iOS Safari (given the device's physical mute switch is deactivated).Persistence Test: Registering 3 subjects, closing the browser tab completely, reopening the application, and registering a 4th subject must automatically set the new ID to 004 without breaking sequence or overwriting historical data.Improper Input Sanitization: Typing symbols, spaces, or hitting the button with an empty input field must trigger a validation warning without iterating the internal counter or transmitting faulty DTMF frames.
